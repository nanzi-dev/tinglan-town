# 《听澜镇》春季样板设计规格

## 1. 产品定义

- 中文名：《听澜镇》
- 英文名：Tides of Tinglan
- 项目目录：`tinglan-town`
- 引擎：Godot 4.x
- 脚本：GDScript
- 类型：2.5D 江南水乡生活模拟 RPG
- 首发平台：Windows、Linux
- 首发语言：简体中文
- 首个完整里程碑：春季 14 个游戏日样板

玩家是刚搬入听澜镇的新居民。镇子在玩家不干预时也会持续运转：十名居民按照日程、需求、关系和正在进行的事务行动。玩家可以观察居民、加入对话、发布委托、送礼、采集、参与社区项目和春日节庆，并通过可靠的行动逐步成为社区的一员。

游戏的核心体验不是经营效率或数值优化，而是“在一个会自行生活的社区中形成真实的熟悉感”。因此关系数值不直接展示，NPC 的行为必须可解释，重要事件必须可追溯，模型不可越过游戏规则创造结果。

## 2. 范围与非目标

### 2.1 春季样板必须包含

1. 一张连续室外镇区，包含河道、桥、主街、集市、码头和可采集区域。
2. 十个可进入室内：玩家住所、茶馆、杂货铺、诊所、工坊、书屋、社区中心和三处居民住宅。
3. 十名有固定身份、能力、关系、日程与个人诉求的核心 NPC。
4. 键盘移动、点击寻路、场景切换、交互提示和基础背包。
5. NPC 近场实时模拟与离屏粗粒度模拟。
6. 场景气泡交谈、旁听、申请加入、自由输入、情境动作和可展开会话记录。
7. 居民委托、社区项目、玩家发布任务三类公告板内容。
8. 约二十个结构化任务模板。
9. 一个完整社区项目“修复听雨桥”，包括提案、筹资/筹物、投票、施工和落成结果。
10. 一场“上巳水灯会”节庆。
11. 至少两名明确成年的可恋爱 NPC，恋爱内容保持克制并建立在信任和长期互动上。
12. 检查点、幂等事件日志、崩溃恢复、最多三个游戏日的离线补算与“镇上近况”摘要。
13. 本地 Memoria 不可用时的明确状态、自动重试和完全可玩的本地降级。
14. 固定随机种子的十名 NPC、十四日确定性模拟。

### 2.2 明确不包含

- 战斗。
- 自由建造。
- 复杂农业。
- 重度商店经营。
- 模型直接移动角色、创建物品、修改时间、发放奖励或判定任务完成。
- 在春季样板阶段同时制作夏、秋、冬和年度结局。

后续季节必须作为独立内容包接入相同模拟接口，不修改春季核心规则。

## 3. 方案比较与决策

### 3.1 方案 A：Godot 权威模拟，Memoria 负责社交认知

Godot 持有世界时间、位置、寻路、需求、物品、任务状态、奖励和完成条件。Memoria 只接收 Godot 提供的合法候选动作，返回候选动作 ID、社交意图、目标、公开原因代码和台词提示。

优点：

- 世界状态可复现、可测试、可离线运行。
- 模型故障不会破坏玩法。
- 幂等边界清晰，存档恢复容易验证。
- 与需求中“模型不得改写世界”完全一致。

缺点：

- Godot 需要实现较完整的规则与本地回退。
- 社交表现与世界规则之间需要稳定契约。

### 3.2 方案 B：Memoria 主导 NPC 行为

Memoria 根据完整世界状态决定 NPC 行动，Godot 负责执行。

优点是表面上的 NPC 自由度较高；缺点是成本、延迟、不可复现、非法动作和恢复复杂度都显著增加，与验收要求冲突。

### 3.3 方案 C：单独的无头模拟服务

使用 Python 或 Go 建立独立世界模拟服务，Godot 只做显示与输入。

优点是服务端测试方便；缺点是引入第三个运行时、双重状态同步和额外部署成本。单机生活模拟样板不需要这一复杂度。

### 3.4 最终决策

采用方案 A。Godot 是世界事实的唯一权威；Memoria 是可替换、可降级的社交认知协作者。所有跨边界消息携带 `request_id` 和 `tick_id`，所有改变世界的结果都由 Godot 对合法动作进行本地验证后提交。

## 4. 总体架构

### 4.1 Godot 分层

`core` 层是不依赖场景树表现的确定性规则：

- `WorldClock`：一天 24 分钟现实时间，维护季节、日、时、分、暂停和时间倍率。
- `SimulationScheduler`：分配近场实时 tick、离屏粗 tick 和社交批处理预算。
- `UtilityEvaluator`：依据需求、日程、关系、任务和上下文给合法动作评分。
- `TaskBoard`：管理居民委托、社区项目、玩家任务、期限、状态和奖励结算资格。
- `RelationshipLedger`：保存好感、信任、戒备与近期原因，对 UI 只输出关系阶段。
- `SaveCoordinator`：写入原子检查点和追加式事件日志，执行幂等恢复与离线补算。

`actors` 层连接规则与游戏角色：

- `AgentController`：生成合法候选动作，选择本地最高效用动作或应用 Memoria 返回的合法候选。
- `PlayerController`：键盘移动、点击寻路、交互与场景入口。
- `NeedsComponent`：饥饿、精力、社交、舒适和职责压力。
- `ScheduleComponent`：按工作日、节庆日和个人事件提供时间段安排。

`services` 层处理外部或跨功能协作：

- `MemoriaClient`：异步 HTTP、超时、重试、请求去重和健康状态。
- `ConversationManager`：旁听、加入许可、线程、自由输入、情境动作和日志。
- `EventBus`：发布结构化领域事件，避免 UI、角色和存档相互直接依赖。

`presentation` 层负责地图、角色表现和 UI，不决定世界结果。

### 4.2 Memoria 分层

Memoria 新增独立的 `/api/v1/game` 路由，不改动现有 Web 前端：

- Pydantic 契约模型。
- 世界绑定存储与服务。
- 游戏智能体决策服务。
- 幂等社会事件投影服务。
- 公告板任务响应服务。
- 与现有人格、记忆、关系、事件和故事服务的适配器。

Memoria 的游戏接口不直接依赖 Godot 文件格式。它只处理版本化 JSON 契约。

### 4.3 跨仓库隔离

`tinglan-town` 在当前目录直接开发，不额外创建分支。

修改 `/home/nanzi/PY3/Memoria` 前必须创建独立分支，并通过单独 Git worktree 开发，以免切换或污染 Memoria 当前工作目录及其未提交改动。分支名使用 `feat/tinglan-game-api`，工作树使用仓库外路径 `/home/nanzi/PY3/Memoria-tinglan-game-api`。

## 5. 数据契约

所有标识符为非空字符串；所有时间同时携带整数 `tick_id` 与结构化世界时间。跨进程请求必须有 UUID 格式 `request_id`。

### 5.1 WorldBinding

```json
{
  "request_id": "uuid",
  "world_id": "tinglan-world-01",
  "save_id": "slot-01",
  "story_id": "memoria-story-id",
  "player_id": "player",
  "revision": 1
}
```

相同 `world_id + save_id` 的重复绑定请求必须返回同一绑定。只有显式提供下一修订号才能更新。

### 5.2 AgentSnapshot 与 AgentDecision

Godot 为每名需要社会决策的 NPC 生成 `AgentSnapshot`，其中包括：

- `character_id`
- 世界时间和位置
- 归一化需求
- 当前目标
- 可用时间
- 附近角色
- 相关任务
- 合法候选动作数组

每个候选动作至少包含 `candidate_action_id`、动作类型、目标、预计时长和公开约束。Memoria 返回：

```json
{
  "character_id": "shen-yan",
  "candidate_action_id": "talk:lin-xi:tea-house",
  "social_intent": "repair_trust",
  "target_character_id": "lin-xi",
  "target_task_id": null,
  "reason_code": "recent_misunderstanding",
  "dialogue_hint": "先问她昨日是否平安，不急着解释。"
}
```

Godot 必须拒绝不在候选集合内的 ID，并选择本地最高效用合法动作。

### 5.3 SocialEvent

事件字段：

- `event_id`：稳定、全局唯一。
- `request_id`
- `tick_id`
- `event_type`
- `participants`
- `world_time`
- `structured_result`
- `source_action_id`

支持的首阶段事件类型为 `encounter`、`promise`、`task_accepted`、`task_completed`、`project_vote`、`relationship_changed` 和 `conversation_joined`。重复 `event_id` 必须返回原处理结果，不得重复写入记忆或关系。

### 5.4 BoardTask

字段：

- `task_id`
- `task_source`：`resident`、`community`、`player`
- `task_type`
- `issuer_id`
- `objective`
- `location_id`
- `deadline_tick`
- `reward`
- `completion_rules`
- `description`
- `status`
- `assignee_ids`

完成条件只由 Godot 的结构化规则判断。自由描述不参与完成判定。

### 5.5 ConversationContext

字段：

- `conversation_id`
- `participant_ids`
- `topic`
- `occasion`
- `player_listening`
- `join_status`
- `memoria_thread_id`
- `transcript`

`join_status` 为 `listening`、`requested`、`accepted` 或 `declined`。玩家只有在 `accepted` 后才能发送自由输入或情境动作。

### 5.6 OfflineCatchupResult

字段：

- `from_tick`
- `to_tick`
- `capped_days`
- `key_events`
- `task_changes`
- `relationship_changes`
- `town_digest`

离线补算只使用确定性本地规则，最多推进三个游戏日。Memoria 可在恢复后润色摘要，但润色失败不影响结果。

## 6. 时间、模拟与模型预算

一个游戏日等于 24 分钟现实时间，即现实 1 秒等于游戏 1 分钟。默认每游戏 10 分钟执行一次逻辑 tick：

- 玩家视野与邻近区域内 NPC：每物理帧平滑移动，每逻辑 tick 更新需求和目标。
- 离屏 NPC：每游戏 30 分钟合并结算一次。
- 重要社交决策：按同一 tick 批量发送，目标为每个游戏日 30 至 60 次批量请求。
- 玩家主动对话：不计入后台预算。

固定随机种子由 `world_id`、`save_id` 和用户可见种子共同派生。所有随机选择必须使用保存于世界状态中的单一可序列化 RNG 状态，禁止直接调用未受控的全局随机函数。

## 7. 世界与视觉设计

### 7.1 室外镇区

镇区采用正交等距镜头。河道从西北流向东南，把空间分成生活、商贸和工艺三个可辨识区域：

- 西岸：玩家住所、居民住宅、菜圃和采集小径。
- 中央：听雨桥、茶馆、杂货铺、公告板和小集市。
- 东岸：诊所、工坊、书屋、社区中心和码头。
- 南侧水湾：上巳水灯会场地。

地图应形成短环路，玩家从任一主要地点步行到相邻功能点不超过一个游戏小时。桥与河岸作为自然地标，不使用迷宫式阻挡。

### 7.2 美术方向

- 简化 3D 建筑和地形，轮廓明确，材质偏手绘。
- 2D 角色精灵始终面向相机，使用四方向行走与待机动画。
- 配色由青灰瓦、白墙、春柳绿、木色、朱红印章色共同构成，避免单一青蓝色界面。
- UI 使用纸、木、墨的材质暗示，但保持平整、清晰和高对比，不使用大面积仿古纹理。
- 卡片圆角不超过 8 像素；主要操作具有图标、文字和键盘焦点。

### 7.3 HUD

常驻 HUD 只展示：

- 左上：春季日期、时间、天气。
- 右上：Memoria 连接状态和可恢复错误入口。
- 下方：当前交互提示与快捷物品栏。
- 右侧按需展开：当前任务摘要。

公告板、对话、背包和关系档案使用全屏或宽面板视图，不在游戏画面上堆叠多个嵌套卡片。

## 9. 任务、社区项目与节庆

二十个模板分为：

- 采集 5 个。
- 递送 4 个。
- 探访 3 个。
- 修缮 3 个。
- 调查 2 个。
- 社交承诺 2 个。
- 节庆准备 1 个。

每个模板均提供结构化目标、地点、期限、奖励和完成规则，以及可由 Memoria 润色的自由描述。

“修复听雨桥”社区项目分五阶段：

1. 陆桥提出结构隐患。
2. 玩家和居民收集木料、石灰与资金。
3. 社区中心召开投票，NPC 根据人格、关系和证据选择支持、反对或弃权。
4. 项目达到阈值后施工，影响两天通行路线。
5. 落成后缩短通行距离，开放桥头纪念对话。

“上巳水灯会”在春季第十二日傍晚举行。节庆结果取决于准备任务完成度、社区项目阶段和玩家与居民的承诺履行情况，但不会产生失败结局；准备不足会改变装饰、出席和对话。

## 10. 对话与本地降级

NPC 之间的普通寒暄使用本地模板和上下文槽位，不调用模型。以下情况可进入 Memoria 批处理：

- 承诺、误会、道歉或关系阶段变化。
- 任务接取、协商、拒绝或退出。
- 社区投票。
- 玩家申请加入 NPC 对话。

Memoria 请求失败时：

1. UI 显示非阻塞连接状态。
2. `MemoriaClient` 使用指数退避重试，单次请求不阻塞主线程。
3. `AgentController` 立即执行本地最高效用合法动作。
4. 对话使用角色本地语料和原因代码生成短句。
5. 待恢复后只补交尚未确认的幂等社会事件，不重放已确认结果。

## 11. 存档与恢复

每个存档槽包含：

- `checkpoint.json`：完整世界快照、RNG 状态和最后应用事件序号。
- `events.jsonl`：只追加的结构化事件。
- `meta.json`：版本、保存时间、世界绑定和内容包列表。

保存流程先写临时文件，刷新后原子替换检查点。奖励、任务推进、记忆投影和项目阶段变化均使用稳定事件 ID。加载时只应用序号高于检查点的事件；相同事件 ID 永不重复生效。

退出时间使用 UTC 保存。重新进入时按真实经过时间换算游戏 tick，限制为三个游戏日，并以同一确定性调度器粗粒度结算。

## 12. 测试与验收证据

### 12.1 Godot 单元测试

- 一天精确等于 1,440 个游戏分钟。
- 暂停时世界时间、需求和日程不推进。
- 相同种子产生相同的十四日事件摘要。
- 非法 Memoria 候选动作被拒绝并触发本地回退。
- 每个游戏日后台批量请求数保持在 30 至 60 次目标区间。
- 任务完成规则、奖励和项目推进对重复事件幂等。
- 离线补算不超过三个游戏日。

### 12.2 Memoria 契约测试

- 所有 `/api/v1/game` 路由鉴权。
- 请求和响应包含 `request_id`、`tick_id`。
- 世界绑定创建、读取和修订冲突。
- 决策只引用请求中的候选动作。
- 社会事件重复提交不重复写入。
- 公告板响应只返回允许枚举。
- 模型超时、异常或非法输出触发可预测回退。

### 12.3 端到端测试

- 旁听 NPC 对话并申请加入。
- 玩家发布任务，NPC 接取、完成并领取一次奖励。
- 听雨桥项目提案、投票、施工和落成。
- 上巳水灯会按准备度展示不同结果。
- 保存、强制退出、恢复后状态一致。
- 离线补算产生结构化变化和“镇上近况”。
- Memoria 不可用时仍可完成上述本地流程，并能在恢复后同步事件。

### 12.4 构建验收

- Linux 可执行构建可启动并进入新游戏。
- Windows 可执行构建产物成功导出。
- 两个平台均可连接已有 Memoria 地址；未连接时显示可恢复错误。
- 无头模式能够执行十四日模拟并以退出码表示通过或失败。
